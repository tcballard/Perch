# v0.0.2 attention layer

Status: recommended design direction; planning only
Decision: hybrid ambient summary plus accessible waiting stack

## Product problem

The v0.0.1 diagnostic roster presents every observed session as a peer. Perch’s
actual job is to answer, in order: whether anything needs a person; what needs
them, why, and for how long; then what else is happening.

## Accepted hierarchy

1. Menu-bar bird and exact waiting count.
2. Compact waiting-first attention panel.
3. Optional expanded cross-provider navigator in v0.0.3.
4. Ambient/spatial representation only when it improves recognition.

Projects are primary. Provider/session is secondary. Working sessions provide
context; waiting sessions drive the interface.

## Information architecture

- Header: Perch, exact waiting count, Refresh.
- Ambient summary: one non-interactive, accessible summary of waiting, working,
  idle, and unknown/stale agents.
- Attention stack: project, normalized required action, relative duration,
  provider/confidence metadata, and Focus or Focus unavailable.
- Secondary mode: All activity, containing every current normalized session.
- Footer: local observation, Quit.

The attention stack owns tab order and actions. Raw prompts, transcript content,
and content-derived paths never enter the presentation model.

## Interaction-model comparison

### Attention inbox

A compact waiting-only grouped stack.

- Fastest identification and strongest linear keyboard/VoiceOver behavior.
- Predictable at 8–20 observed agents because only waits appear by default.
- Makes unknown/stale evidence easy to represent conservatively.
- Risks feeling like another notification list and says little about calm
  background activity.

### Ambient perch

Spatial agent presences occupy calm, resting, uncertain, or attention areas;
waiting agents reveal handoff cards.

- Distinctive and expressive with zero to three agents.
- Can make a transition into waiting spatially obvious without opening rows.
- Crowds quickly at 8–20 agents, complicates keyboard navigation, and risks
  decorative motion competing with work.
- Needs a complete static and reduced-motion equivalent.

### Hybrid

A shallow ambient summary sits above a bounded waiting-only stack.

- Keeps the inbox’s fast scanning and linear accessibility.
- Adds useful context and Perch character without making ornaments actionable.
- Aggregates background states at high density while preserving exact waits.
- Costs vertical space and requires a strict rule that the first handoff remains
  visible without scrolling.

## Scenario evaluation

| Scenario | Attention inbox | Ambient perch | Hybrid recommendation |
| --- | --- | --- | --- |
| 0 waiting | “Nothing needs you”; background hidden | Expressive empty/resting scene | Calm empty message plus quiet state summary |
| 1 waiting | One immediate row | One clear presence/card | One row plus small redundant presence |
| 3 waits | Strong grouped stack | Attention area starts to crowd | Three bounded rows; ambient count/positions |
| 8 observed | Predictable | Small individual presences required | Individuals may remain, but non-waits aggregate |
| 20 observed | Unchanged unless many waits | Visual noise without heavy aggregation | Aggregate background counts; keep exact wait rows |
| Unknown/stale | Neutral All activity rows | Easy to over-dramatize | Neutral/dashed outside attention area and in All activity |
| Focus available | Clear Focus button | Card action can be discoverable | Clear row action; ambient stays non-interactive |
| Focus unavailable | Honest text state | Risks looking broken spatially | Textual unavailable state in row |
| Menu popover | Best fit | Highest crowding risk | Fits at 360×420–480 with shallow summary |
| Keyboard/VoiceOver | Native linear model | Complex if every presence acts | Linear stack; single ambient summary element |
| Contrast/motion | Straightforward | Requires careful alternative | Semantic static summary; motion strictly optional |
| Recognition time | Baseline fastest | Variable | Expected near-baseline; validate against inbox |

## States

- Zero waits with activity: “Nothing needs you” plus quiet aggregate context;
  do not list working/idle sessions by default.
- No sessions after initial refresh: “No active agents. Perch is watching enabled
  providers.”
- Initial loading: stable layout and “Checking local agents…”; never inherit
  waiting without fresh evidence.
- Adapter failure: clear waits immediately; All activity may show Unknown ·
  Stale for up to 15 seconds.
- Unsupported version: “Provider update needs validation” in All activity; it
  never appears as an urgent handoff.
- Empty All activity: explain that no active sessions are currently observed.

## Action categories

Use a closed mapping only:

| Validated evidence | Category |
| --- | --- |
| General structured human input | Input required |
| Explicit permission/approval | Permission required |
| Explicit bounded selection | Choice required |
| Explicit review request | Review required |
| Ambiguous/unsupported evidence | Unknown, not waiting |

## Keyboard and accessibility

- Tab/Shift-Tab traverses Refresh, mode selection, waiting focus actions, All
  activity actions, and Quit in visual order.
- Return/Space activates the focused control. Command-R refreshes; Command-1
  selects Attention; Command-2 selects All activity; Command-Q quits.
- Ambient presences are hidden as individual accessibility elements. One summary
  announces, for example, “8 agents: 2 waiting, 3 working, 2 idle, 1 unknown.”
- Each handoff groups project, action, duration, provider, confidence when useful,
  and focus availability into one coherent announcement.
- Text and symbols accompany semantic color. Use system colors/materials for
  light/dark mode and increased contrast.

## Reduced-motion equivalent

All essential state is present in static position, label, symbol, and count.
With Reduce Motion, agents change appearance/position without animation; no
breathing, bouncing, path movement, or timed reveal remains. The waiting stack
is unchanged.

## Recommendation and rationale

Use the hybrid for v0.0.2, constrained by three rules:

1. The first waiting handoff is visible without scrolling.
2. The ambient layer is non-interactive and has one accessibility summary.
3. At eight or more agents, background states aggregate and never compete with
   the actionable stack.

This serves “which agent needs me?” better than a conventional thread list while
retaining the inbox as the semantic, keyboard, and high-density foundation.

## Rejected alternatives

- Generic all-thread list: weakens attention priority.
- Pure ambient perch as default: novelty and density risks exceed its benefit.
- Provider-first sections: contradict project-first user organization.
- Notch/always-on-top overlay: unnecessary complexity for v0.0.2.
- Provider controls, replies, or approvals: violates observe-and-focus.

## Open owner questions

- No blocking product decision is required for planning. Implementation should
  use the recommended in-panel All activity mode and keep the 360-point width,
  permitting up to 480 points height only if the first handoff remains visible.
- Incorporating third-party code/assets would materially change licensing and
  requires explicit owner approval. The recommendation is no reuse.

## Concrete acceptance criteria

- Deterministic fixtures cover 0 waiting, 1 wait, 3 waits, 8 observed agents,
  20 observed agents, stale/unknown, and both focus states.
- Attention mode contains only waiting sessions and preserves exact count.
- All activity contains all sessions and cannot promote ambiguous state.
- Project and action lead every handoff; provider remains secondary.
- Keyboard-only and VoiceOver passes complete without interacting with ambient
  ornaments individually.
- Reduced Motion and increased contrast communicate identical information.
- Five controlled transitions, provider isolation, sub-three-second visibility,
  and no-network verification remain green.
